# 📋 Support Ticket API

Разработка продвинутого backend-приложения для службы поддержки. Система позволяет управлять клиентами, обращениями (тикетами), вести общение в реальном времени и мгновенно получать аналитику благодаря кэшированию.

---

## 🛠 Технологический стек

| Технология | Описание |
|---|---|
| Golang | Основной язык разработки |
| PostgreSQL | База данных для хранения сущностей и связей |
| Redis | Кэширование аналитики и хранение сессий |
| REST API + WebSockets | CRUD через REST, чат через сокеты |
| JWT | Безопасная авторизация и контроль доступа |
| Docker + Compose | Поднятие БД, Redis и API одной командой |
| Git | Контроль версий |

## ✨ Сущности системы

### 1. 👤 Клиенты (Clients)
* Являются пользователями, которые создают обращения.
* Через ролевую модель (RBAC) могут просматривать только свои тикеты.
* Доступны операции: создание, получение по ID, получение списка, обновление, удаление.
* Валидация ФИО: обязательное поле от 3 до 100 символов.
* Валидация Email: обязательное поле, уникальное в БД, стандартный формат.

### 2. 👥 Сотрудники поддержки (Agents)
* Являются специалистами, которые решают проблемы клиентов.
* Видят все открытые тикеты в системе.
* Могут забирать свободные тикеты, писать ответы клиентам и закрывать тикеты.
* Содержат поля БД: ID, ФИО, Email, Department, created_at.

### 3. 🎫 Обращения (Tickets)
* Привязаны к клиенту-автору и содержат поле agent_id (кто решает проблему).
* Жизненный цикл включает переходы статусов: open -> in_progress -> resolved.
* Тема (Subject) ограничена от 5 до 150 символов.
* Описание проблемы (Description) ограничено от 10 до 2000 символов.

### 4. 💬 Комментарии (Comments)
* Выступают инструментом для общения внутри тикета между клиентом и агентом.
* Сообщения отсортированы по времени создания.
* Содержат поля БД: ID, ticket_id, author_id, author_type (client или agent), text, created_at.

## 🚀 Продвинутая архитектура

### 🔐 Авторизация и Роли (JWT)
* Добавлена генерация и валидация Access и Refresh JWT-токенов.
* Эндпоинты защищены с помощью Middleware.
* Реализовано строгое разделение прав: клиент видит свои тикеты, агент видит открытые тикеты, админ удаляет пользователей.

### ⚡ Кэширование и Фоновые задачи (Redis)
* Подключен Redis для устранения нагрузки на PostgreSQL тяжелыми запросами COUNT и AVG.
* Фоновый Worker на горутинах раз в минуту пересчитывает статистику.
* Эндпоинт аналитики отдает данные мгновенно из кэша.

### 🔄 Real-time взаимодействие (WebSockets)
* Поднято WebSocket соединение для чата в тикете.
* Сервер мгновенно пушит новые ответы подключенному клиенту.
* Реализован паттерн Event-Driven, заменяющий постоянное обновление страницы.

## 📊 Статистика и SLA

| Метрика | Описание |
|---|---|
| Всего тикетов | Общее количество созданных обращений |
| Решённые тикеты | Количество тикетов со статусом resolved |
| Процент успешности | Доля решенных тикетов (решенные / всего × 100%) |
| Среднее время ответа | Расчет времени первого ответа агента и среднего времени до закрытия |
